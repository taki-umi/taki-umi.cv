### 1. 旅行代理店向け システム開発支援業務

項目 | 内容
--- | ---
**在籍期間** | 2022/08/01 ~ ※参画中
**チーム規模** | 10 人ほど
**言語・フレームワーク** | Java/ JavaScript/ Spring Boot
**インフラ** | AWS
**ツール** | GitHub/ Slack/ A5m2/ Docker
**プロダクトが属する業界** | 旅行業界
**プロダクト概要** | 海外ホテル、海外アクティビティの商材登録および予約管理をするためのシステム開発
**チームでの役割** | システムの要件定義、設計、開発、保守運用

#### 1.1 チームの課題

チーム内で管理しているソースコードが、以下の課題を抱えていました。

* 重複コードが複数箇所存在し、影響箇所を特定するのに時間がかかった。
* 1000行を超える巨大なクラスファイルが複数存在し(主にServiceクラス)、コードの見通しが悪く、修正箇所を特定するのに時間がかかった。
* 上記の結果、仕様変更や機能改修に平均3~5日を要し、リファクタリング箇所でバグが発生するなど、開発効率が著しく低下していた。

#### 1.2 課題に対するチャレンジ

このままでは開発効率がさらに悪化すると考え、「ソースコードの変更をしやすくする」ことを目標に、以下の段階的なアプローチで取り組みました。

1. JUnitを用いたユニットテストの充実

* テストコードの作成にあたり、JUnitを導入し、テストケースの作成を効率化した。
* 各クラスの主要なメソッドに対して、網羅率80%以上のテストコードを作成した。
* ユニットテストをPR時、ブランチにマージ時にCIで実行できるようにした。
* テストケース名(メソッド名)もどのようなテストかを明確にするため、テストケース名ルールを定めた。

```
例）

// NG
// これだと、どのようなテストか具体的にわからない
@Test
public void testValidate_NG() {
    // テストコード
}

//
// 上記のような具体的でないテスト名を以下の通り修正していった
//

// OK
// これだと、どのようなテストか具体的にわかる
@Test
public void test_validate_予約番号が指定されていない場合はエラー() {
    // テストコード
}
```

2. 各 API の仕様を担保するためのインテグレーションテストコードの追加

* ユニットテストだけでは各APIの仕様を担保できないため、結合テストを目的としたインテグレーションテストコードを作成した。
* インテグレーションテストでは、各APIの試験観点を洗い出し、それぞれのAPIのテストケースとしてテストコードを作成した。
* ユニットテストとは別でインテグレーションテスト用のgradleタスクを定義し、CIで実行できるようにした。

#### 1.3 チャレンジに対する結果・成果

上記の通り、テストコードを充実させたことで、心理的安全性の高いソースコードを実現し、以下の成果を上げました。

* デグレ発生率を大きく削減し、手戻りを大幅に減らすことができた。
* リファクタリングによって、コード行数を平均300行以下にするようにし、コードの可読性を向上させた。
* チームメンバーからは「以前は変更に不安を感じていたが、テストコードがあることで安心して変更できるようになった」という声が寄せられた。

### 2. 認証認可システム開発支援業務

※上記、旅行代理店向けシステム開発支援業務と並行で対応。

項目 | 内容
--- | ---
**在籍期間** | 2024/05/01 ~ ※参画中
**チーム規模** | 2 人
**言語・フレームワーク** | Go/ React/ Gin/
**インフラ** | AWS
**ツール** | GORM/ GitHub/ Slack/ Docker
**プロダクトが属する業界** | 旅行業界
**プロダクト概要** | 旅行代理店会社の事業で扱う内製システムの認証認可機能を担うシステムのメンテナンス
**チームでの役割** | システムの追加要望改修・保守運用

#### 2.1 チームの課題

システムの途中引き継ぎを行ったものの、以下の情報が不足しており、開発に支障をきたしていました。

* システムのアーキテクチャ、ローカル環境構築手順などのドキュメントが一切存在せず、キャッチアップに時間がかかっていた。
* DBドキュメントが存在しないため、DBの情報を一元化できておらず、システムの理解・把握が難しかった。

#### 2.2 課題に対するチャレンジ

この状況を改善するため、以下の取り組みを実施しました。

1. READMEの作成

* 以下のとおり、開発に必要な情報を網羅したREADMEを作成しました。
  * システムの概要
  * 使用言語・フレームワークなどのバージョン情報
  * ディレクトリ構成
  * ローカル環境構築手順
  ※ デバッグ方法
* 特に、「デバッグ方法」について詳細な情報を記載することで、自信含め後任者がスムーズに開発に着手できるように工夫しました。

2. DBドキュメント作成の文化定着

* DBドキュメント作成ツールである[tbls](https://github.com/k1LoW/tbls)を導入し、ER図、テーブル定義書、カラム定義書など、必要なドキュメントを自動生成する仕組みを構築しました。
* READMEにtblsの使い方を記載し、DBドキュメントの重要性を説明することで、DBドキュメント作成の文化を定着させました。
* READMEにDBドキュメントの運用方法を記載することで、ドキュメントが常に最新の状態に保たれるようにしました。

#### 2.3 チャレンジに対する結果・成果

これらの取り組みにより、以下の成果を上げることができました。

* READMEを作成したことで、後任者が開発を引き継ぐ際の学習コストを大幅に削減しました。
* DBドキュメントを作成したことで、システム全体の理解が深まり、機能の改修をスムーズに進めることができました。
